Uit voorgaand onderzoek is gebleken dat Firebase offline data persistence ondersteunt en dit als oplossing kan dienen voor het probleem van netwerk gebrek in vluchtelingen kampen bij het gebruik van de app. Deze bevinding zal verder onderzocht worden door het maken van prototypes waarin deze techniek getest zal worden op de verschillende use cases van de app.

Welke tussenoplossingen zijn er voor de mogelijkheden van de netwerk situaties?


Aanpak

Door middel van de onderzoeksstrategie workshop met behulp van de onderzoeksmethode prototyping, ga ik de verschillende offline use cases van de app testen. Dit doe ik door de app op een Iphone device te runnen en hierbij verbreek ik het de internet connectie. Daarnaast houd ik de Cloud Firestore database hier langs om te bekijken wanneer er aanpassingen plaats vinden op basis van de inputs op het device. Op basis van deze resultaten zal er bekeken worden hoe deze resultaten invloed hebben op de offline use cases en of er verder onderzoek nodig is voor een oplossing op de onderzoeksvraag.



Resultaat

Er zullen voornamelijk 3 use case flows zijn die essentieel zijn voor de werking van de app. Deze 3 flows zijn hieronder beschreven.

1. Booking & return flow, gebruikers moeten sportspullen en andere items kunnen uitlenen en terug brengen via de app.
2. Member flow, gebruikers moeten een lijst kunnen zien van de members en de uitleen activiteiten die hierbij horen. Er moeten nieuwe members aangemaakt, verwijderd en aangepast kunnen worden.
3. Stock flow, gebruikers moeten een lijst kunnen zien van de stock items en de hoeveelheden hiervan, deze stock items moeten in de app ook aangepast kunnen worden

Deze use case flows moeten bruikbaar zijn voor de gebruiker van de app in zowel online als in offline mode.



1. Aanpassen van een member
In deze prototype wordt het aanpassen van een member offline getest. Links is de Cloud Firestore database te zien en rechts de mobile app. Hier wordt het geslacht van de member Harry veranderd van een man naar een vrouw.

Resultaat
Zoals is te zien in de video wordt de aanpassing offline gemaakt, maar de UI wordt wel geupdate. Dit komt doordat de app offline tegen een cached versie van de database praat. Zodra het device weer een internet connectie heeft wordt het bewaarde request verstuurd naar de database. Het nadeel hiervan is zodra er meerdere gebruikers offline het geslacht van Harry aanpassen, dat het laatste request dat verzonden wordt "wint". Wanneer gebruikers langere periodes zonder een netwerk verbinding zitten kan dit verlies van data requests veroorzaken.



2. Aanmaken van meerdere bookings
In deze prototype wordt het aanmaken van meerdere bookings getest. Ook hier is links de Cloud Firestore database te zien en rechts de mobile app. 3 items worden achter elkaar geboekt terwijl het device geen netwerk verbinding heeft.

Resultaat
De items worden offline geboekt maar de aangemaakte bookings zijn nog niet in UI van de returns te zien. Zodra het device weer een netwerk verbinding heeft worden de bookings automatisch toegevoegd aan de Cloud Firestore database. Hiermee wordt de UI van de app ook gelijk geupdate. Wat opvallend hierbij is, is dat de UI bij het aanmaken van een nieuw document, niet update terwijl die dit wel doet bij het aanpassen van een document.



3. Verwijderen van een booking
In deze prototype wordt het verwijderen van een booking getest. Dit wordt vanuit de member detail pagina gedaan, in het scherm hierboven zijn alle members te zien en wanneer een member een booking heeft die al terug gebracht had moeten zijn, wordt dit aangegeven met het rode label "Due".

Resultaat
De booking van Jane mood wordt offline verwijderd. Wanneer deze wordt verwijderd wordt de UI ook aangepast, daarnaast verdwijnt het rode "Due" label ook. Zodra de netwerk verbinding weer aanwezig is wordt het document ook in de Cloud Firestore database verwijderd. Opvallend hier is dat bij het verwijderen en aanpassen van documenten, de UI mee veranderd maar dat dit niet gebeurd bij het aanmaken van een nieuw document.



Conclusie

Uit de resultaten is gebleken dat wanneer het device geen netwerk verbinding heeft, er een kopie van de Cloud Firestore database wordt gemaakt en verdere aanpassingen hiermee interacteren. Zodra het device weer een netwerk verbinding heeft worden de opgeslagen requests van de user naar de database verstuurd en synchroniseerd het device met de Cloud Firestore database. Hierbij viel op dat de UI bij het aanpassen en verwijderen van documenten offline werden geupdate maar dit gebeurde niet bij het aanmaken van nieuwe documenten.
